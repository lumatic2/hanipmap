# API 계약 (동결)

> 파트 사이에 주고받는 데이터 형식. 2026-07-18 확정.
> **개발 중 변경 금지.** 바꾸고 싶으면 구현 전에 통합 담당과 논의한다.
> 화면 파트는 서버가 준비되기 전까지 아래 "응답 예시"를 가짜 데이터로 써서 개발을 시작하면 된다.

## 1. 사진 파싱 — 파트 B가 호출

```
POST /api/parse-menu
요청: { "image": "<base64 문자열>", "filename": "menu.jpg" }
응답: { "items": [ { "name": "김치찌개", "price": 8000 },
                   { "name": "된장찌개", "price": 7000 } ] }
실패: { "error": "이미지를 읽을 수 없습니다", "code": "PARSE_FAILED" }
```

## 2. 검색 해석 — 파트 C가 호출

```
POST /api/parse-query
요청: { "query": "과선배가 사는 날 2만원 둘이" }
응답: { "budget": 10000, "walkMax": 10, "tags": ["회식"],
        "reason": "2만원을 둘이 나누면 1인 1만원", "source": "solar" }
```

- 항상 200으로 응답한다. 해석 실패 시 `source: "fallback"`과 기본값을 준다. 화면 쪽 에러 처리 불필요.
- `budget`은 1인 기준 원 단위, `walkMax`는 도보 분.

## 3. 행동 기록 — B·C·D 각 화면에서 한 줄 호출

```
POST /api/events
요청: { "place_id": "p123", "action": "select" }
응답: { "ok": true }
```

- `action`은 `select`(카드 선택) · `detail`(상세 진입) · `navigate`(길찾기 클릭) · `report`(제보 제출) 중 하나.
- 로그인 처리는 서버가 알아서 한다(익명 계정). 화면은 위 두 필드만 보내면 된다.

## 4. 추천 — 파트 D가 호출

```
GET /api/recommend
응답: [ { "place": { "id": "p123", "name": "명륜진사갈비", "lat": 37.58, "lng": 126.99 },
          "score": 0.87,
          "reason": "지난주 국밥을 두 번 고르셨고 오늘 예산 안입니다" } ]
```

- 본인 기록은 서버가 자동으로 참조한다. 기록이 없는 신규 사용자에게도 인기 기준 추천으로 항상 200을 준다.

## DB 테이블 (파트 A 소유)

| 테이블 | 내용 |
|---|---|
| `places` | 식당: 이름, 좌표, 주소, 카테고리 (지도 API로 일괄 등록) |
| `menus` | 메뉴: 식당 id, 이름, 가격, 상태(승인 대기/승인) |
| `cafeteria_menus` | 학식: 날짜, 식당, 메뉴, 가격 (자동 수집) |
| `user_events` | 행동 기록: 사용자 id, 장소 id, 행동, 시각 |

스키마 파일은 `db/schema.sql`로 관리한다.
